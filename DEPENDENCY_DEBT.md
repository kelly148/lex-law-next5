## DD-001 — drizzle-orm generatedAlwaysAs() API

**Deferred from:** Phase 4b (D.1.2 resolution)
**Target resolution:** Future cleanup (post-Phase 7)

Upgrade drizzle-orm to 0.32+ and migrate raw-SQL generated columns (`activeMatterKey`,
`activeSessionKey`) to native `generatedAlwaysAs()` builder API. Deferred from Phase 4b per
D.1.2 resolution — upgrade mid-build carries cascade risk across four merged phases
disproportionate to the compile-time gap being closed.

**Compensating controls in place (Phase 4b):**

1. Raw SQL migration uses MySQL/TiDB-compatible `GENERATED ALWAYS AS (...) STORED` syntax.
2. `schema.ts` column declarations carry explicit DO-NOT-WRITE comment blocks referencing R10,
   Ch 4.10, and Ch 4.8.
3. The Zod Wall treats these columns as read-only (no write path in any procedure).
4. Regression tests assert that any INSERT or UPDATE supplying an explicit value for either
   generated column is rejected by the database engine.
5. This file is included in the Phase 7 Known-Issue List (HANDOFF.md).

**Upgrade path:**

```bash
pnpm add drizzle-orm@^0.32
# Replace activeMatterKey and activeSessionKey declarations in schema.ts
# with .generatedAlwaysAs(sql`...`, { mode: 'stored' })
# Drop the raw-SQL ADD COLUMN / ADD INDEX statements from the migration
# Run: pnpm drizzle-kit generate && pnpm drizzle-kit migrate
```

## DD-003 — Migration SQL generation environment mismatch

**Discovered during:** Deployment to TiDB Cloud Serverless (D.1.6, 2026-04-25)
**Target resolution:** Before next migration is generated and deployed

Migrations generated by `drizzle-kit generate` were generated against a development environment
more permissive than the deployment target (TiDB Cloud v8.5.3-serverless). Specifically,
`drizzle-kit` produced `DEFAULT ('[]')` and `DEFAULT ('{}')` for JSON columns, which TiDB Cloud
rejects in favour of `DEFAULT (JSON_ARRAY())` and `DEFAULT (JSON_OBJECT())` syntax.

**Affected migration file:** `src/server/db/migrations/0000_sparkling_firestar.sql`

| Column | Table | Original (rejected by TiDB) | Fixed |
|---|---|---|---|
| `tags` | `matter_materials` | `DEFAULT ('[]')` | `DEFAULT (JSON_ARRAY())` |
| `preferences` | `user_preferences` | `DEFAULT ('{}')` | `DEFAULT (JSON_OBJECT())` |

**Fix applied:** Both lines patched in `0000_sparkling_firestar.sql` on branch
`lex-next/migration-tidb-compat` (merged to `main`). Pre-verified on TiDB Cloud v8.5.3-serverless
before applying.

**Resolution required for future migrations:**

1. Run `drizzle-kit generate` against TiDB Cloud (or a TiDB-compatible local test instance such
   as TiDB Playground or `tiup playground`), not local MySQL/Postgres.
2. OR add a post-generate normalization script that converts string-literal JSON defaults
   (`DEFAULT ('[...]')`, `DEFAULT ('{...}')`) to expression form (`DEFAULT (JSON_ARRAY())`,
   `DEFAULT (JSON_OBJECT())`).
3. OR document the manual patch step in `DEPLOYMENT.md` so future migrations are reviewed
   before deployment.

**Estimated effort:** 1–2 hours to set up a TiDB-compatible local test environment for
`drizzle-kit`. Discovered during deployment to TiDB Cloud Serverless when migration applied
against the production cluster failed with `ER_PARSE_ERROR`.
